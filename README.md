## **הערות מקדימות:**

    1.
    בחרתי להשתמש ב'קפקא' כי הייתי צריך כלי להעברת הודעות קלות 
    ולא ליצור 'האשים' עם אפשרות לחיפושים מהירים כמו ש'רדיס' היה מאפשר לי
    2.
    כדי להקל על הבנת מבנה הפרוייקט בהמשך
    בחרתי לתת שמות פשוטים לקבצים שמכילים את הקלאסים
    שמגדירים חיבורים וקונפיגורציה ומאפשרים פעולות
    נניח קונסיומר של קפקא יישב פשוט בקובץ בשם
    'kafka'
    וכן על זו הדרך 
    3.
    במקום לשמור קובץ לוגר אחד ולכוון את הסרוויסים 
    להעתיק אותו לתיקיית הדוקר שלהם  או לקרוא ממנו, 
    העדפתי לשים בכל תיקייה מקומית קובץ כזה מראש כדי להקל על הבנת מבנה הפרוייקט

## **DATA_LOADER:**

    טוען מידע מקבצי 
    JSON ו- CSV
    מסדר ומחלץ חלק מהמידע לפי הדרישות ומחלק אותו למידע שאינו 'סכמטי' ולכזה שכן
    שולח את המידע הלא-מובנה לתור בקפקא ואת המובנה טוען לטבלאות ב
    MYSQL DB לפי הדרישות

## **DATA_ANALYZER:**

    צורך את ההודעות ש
    -'data_loader'-
    טען לקפקא, 
    מפריד בין הודעות ה
    -'GPS' וה-'INTERCEPTS'
    עושה ניקוי וניתוח לנתונים לפי דרישת המשימה 
    וטוען אותם לשני אוספי
    -'MONGO_DB' לפי סוג ההודעה


## **END_USER-SERVICE:**

    מחלץ את ההודעות שנשמרו במונגו-
    --אודות החשודים וחשבונות הבנק שלהם השמורים ב--
    SQL-DB
    מצליב הודעות עם 'חשודים-שמורים' ומעשיר אותן במידע נוסף לפי הדרישה,
    ואז טוען את ההודעות המפורטות ל 
    'ELASTICSEARCH'
    בנוסף כתבתי שרת בסיסי שמאפשר למשתמש הקצה לגשת ולחפש טקסט חופשי ב-
    'ELASTICSEARCH' 
    עם 'בוסטינג' לפי הדרישה.


## **LOGGER:**

    במקום להשתמש בספריית לוגר מובנית אתגרתי את עצמי מעט ויצרתי אחד כזה.
    הלוגר מתבסס על כך שכל הסרוויסים כותבים את הודעותיהם לטופיק מסויים בקפקא 
    קובץ הלוגר בכל סרוויס הוא קלאס חיבור לקפקא שגם מעשיר כל הודעה ב
    time-stemp,
    service-name (using "Path(__file__).parent.name" from pathlib library,
    message-type ('info'/'error')
    בצורה הזו אפשר פשוט לגשת לקובץ הלוגים של הסרוויס 
    LOGGER
    ולראות את כל הלוגים של כל סרוויס בפורמט הבא:
    
    #########
    - service-name
    #########
    - time-stemp
    - message-type ('info'/'error')
    - הלוג של הסרוויס ששלח אותו
